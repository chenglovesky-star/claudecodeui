# 记忆管理规范

## 存储位置

记忆存储在**当前项目根目录**（Claude Code 打开的目录）的 `./memory/` 下。不存在时自动创建。

```
<项目根目录>/
└── memory/
    ├── MEMORY.md          ← 索引文件（必须）
    ├── general.md         ← 通用知识
    ├── biz_community.md   ← 社区业务
    ├── biz_feed.md        ← 信息流业务
    └── biz_assistant.md   ← 助手业务
```

## 初始化

第一次使用时，检查 `./memory/MEMORY.md` 是否存在。不存在则创建以下文件：

### MEMORY.md 模板

```markdown
# Memory Index

## 业务索引
| 业务 | 文件 | 操作码数 | 模板数 |
|------|------|---------|--------|

## 通用知识
- [general.md](general.md) — 工具限制、性能经验、跨业务公共表、画像标签

## 通用速查（每次 SQL 执行前检查）

- executeQuery **禁用** `--` 注释，必须去掉
- tags 提取用 `ifly_map_get(tags, 'key')`，tags_json 不可靠

## 读取策略

1. 收到需求 → 读本文件 → 按业务匹配文件
2. 命中业务 → 读对应 biz_*.md 获取操作码+Tag关联+模板
3. 通用知识 → 读 general.md 检查工具限制和公共表
4. 新业务积累 3 条以上知识后 → 创建新的 biz_*.md
```

## 文件分类

- **业务文件**：`biz_<业务>.md`（按业务线分类）
- **通用知识**：`general.md`（工具限制、性能经验、跨业务公共表）
- 新业务积累 3 条以上知识后创建新 `biz_*.md`
- 新知识优先追加到已有文件，而非创建新文件
- **memory 文件不得超过 10 个**

## 不存储表结构

表字段、类型、分区等结构信息不存本地，每次通过 MCP（describeTable）实时查询。

## 何时写入

- 确认了某个业务指标的精确口径
- 发现新的操作码或 Tag 关联关系
- SQL 执行失败且原因有复用价值
- 发现枚举值有陷阱或业务含义

## 何时不写

- 一次性的查询参数（日期范围、具体用户ID）
- 已存在的知识（避免重复）
- 过于琐碎、无法泛化的信息
- 表结构详情（线上为准）

## 更新标准

- **口径纠正** — 发现本地口径与实际不符时立即更新
- **SQL 模板失效** — 执行报错时标记失效并修正
- **操作码变更** — 新增、废弃或含义变化时立即更新
- **Tag 关联变化** — 关联表/字段变化时立即更新
- **踩坑记录过期** — 平台修复后删除

## 中心同步

用户选择「同步到中心」时，调用 `saveSqlGuide` 传入：
- **keyword**：搜索关键词
- **description**：用户原始需求
- **sql**：验证通过的完整 SQL

## 写入后必须同步更新 MEMORY.md 索引

## 第六阶段：记忆评估标准

每次查询完成后，主 Agent 按以下维度逐项评估，任一命中则建议写入：

### 评估维度

| 维度 | 判断标准 | 示例 |
|------|---------|------|
| **口径发现** | 确认了某个指标的精确计算方式（非显而易见的） | UV 必须先按 uid 去重再汇总，不能直接 COUNT(DISTINCT) |
| **表/字段映射** | 发现了需求中业务概念 → 具体表/字段的非直觉映射 | "用户活跃"不在 user 表，而在 behavior_log 表的特定 opcode |
| **操作码关联** | 新发现了 opcode 与业务场景的对应关系 | opcode=12345 对应社区帖子浏览 |
| **Tag 陷阱** | 枚举值含义与字面不符，或存在隐藏约束 | platform='1' 实际是 iOS 而非 Android |
| **仲裁经验** | 两个 Agent 不一致的原因有泛化价值 | 某表的 dt 字段格式是 yyyy-MM-dd 而非 yyyyMMdd |
| **性能教训** | 查询超时或资源溢出，且解决方案可复用 | 该表必须先按 dt 分区裁剪再 JOIN |
| **工具踩坑** | MCP 工具行为与预期不符 | searchTables 对中文关键词效果差，需用英文 |

### 不写入的情况

- 查询结果符合预期，过程无意外
- 所用口径已存在于现有记忆中
- 发现的知识过于特定（如"2026年3月某天数据异常"）
- 仅是日期/参数不同的重复查询

## 记忆回顾与维护机制

### 触发时机

1. **被动触发** — 每次读取记忆时顺便检查：
   - 读到的条目是否仍然正确？（如果本次查询涉及该条目相关的表/字段，顺手验证）
   - 读到的模板是否仍能执行？（如果复用了记忆中的 SQL 模板，执行结果验证）
2. **主动触发** — 当以下条件满足时，在第一阶段额外执行回顾：
   - `MEMORY.md` 中业务文件数 ≥ 8（接近上限 10）
   - 某个 `biz_*.md` 文件超过 150 行
   - 用户显式要求（如"清理一下记忆"）

### 回顾操作

| 检查项 | 操作 |
|--------|------|
| **过时口径** | 如果本次查询发现记忆中的口径与实际不符 → 立即更新 |
| **失效模板** | 记忆中的 SQL 模板执行报错 → 标记 `[⚠️ 待验证]` 或直接修正 |
| **冗余条目** | 同一知识点出现在多个文件中 → 合并到最相关的文件 |
| **低价值条目** | 过于琐碎、从未被引用过的条目 → 删除 |
| **文件膨胀** | 单文件超 150 行 → 拆分或精简（保留高频使用的条目） |

### 淘汰规则（文件数接近上限时）

1. 优先合并相近业务的 `biz_*.md`（如 `biz_feed.md` 和 `biz_recommend.md` 合并为 `biz_content.md`）
2. 将低频业务的核心知识压缩到 `general.md` 的专门小节
3. 删除超过 6 个月未被引用且无踩坑记录的业务文件
4. **删除前必须用 AskUserQuestion 确认**

### 记忆有效性标记

在记忆条目中可选添加元信息（作为行尾注释）：

```
| opcode=12345 | 社区帖子浏览 | tags: page_id, content_type | <!-- verified:2026-03-23 -->
```

- `verified:<日期>` — 最近一次验证通过的日期
- `stale` — 已知过时，待更新
- `unverified` — 从未被实际查询验证过
